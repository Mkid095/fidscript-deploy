  - **Port :25 plain SMTP**: the greeting works perfectly ‚Äî `220 deploy.fidscript.com Stalwart ESMTP at your service`. So Stalwart is alive and speaking SMTP on :25.
  - **Port :465 (SMTPS, `tlsImplicit: true`)**: TLS handshake **succeeds** ‚Äî `openssl s_client` confirms Stalwart presents a self-signed ECDSA cert (`CN = rcgen self signed cert`, TLSv1.3, TLS_AES_256_GCM_SHA384). That is the `"Greeting never received"` error. Not a credentials issue (that would be `535 5.7.8`).

This is decisive: **:25 completes the SMTP exchange** but Stalwart's :25 (the incoming listener) refuses to **relay to external recipients** ‚Äî which is correct for an incoming listener. The two listeners fail in **different, server-side ways**:
    - `submissions` (`:465`) ‚Äî TLS fine, SMTP greeting never sent (Stalwart-side listener bug / misconfig on v0.16.11).
    - `smtp` (`:25`) ‚Äî SMTP fine, but no relay policy for the API's submission (correct for an incoming listener, but means :25 is not a viable submission path).
  - Stalwart container logs (`docker logs fidscript_stalwart --since 15m`) are **empty** ‚Äî the server is silent (no log output for the TLS handshake, the missing greeting, or the relay rejection), which is itself a notable signal about the listener.
  - **Reverted** the compose change ‚Äî :25 is not a real fix (relay denied), so the platform is back to the documented `"Greeting never received"` state on :465, ready for a proper fix.
  - **Root cause and fix path**: this is a **Stalwart-side** issue, not an application bug. The application-side transport configuration (`stalwart-transport.ts` + `smtp-send.service`) is correct for the assumed contract. Options to unblock real outbound delivery: (a) **investigate the Stalwart `submissions` listener** ‚Äî it may need an explicit `auth` / `directory` / `relay` config block, or a properly-registered SMTP submission account (the `admin` directory user may not have implicit submission rights on :465); (b) **try an earlier Stalwart v0.16 patch** (the upgrade to :latest = v0.16.11 may have introduced this ‚Äî v0.16.10 was working at the JMAP level, the SMTP-side regression is possible); (c) **configure Stalwart to use the `queue` for outbound delivery** and submit via the JMAP/api rather than SMTP submission. All three require a Stalwart config change (or a rollback) ‚Äî NOT an application change. Recorded here so this is not lost.
